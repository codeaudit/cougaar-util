COUGAAR Society Startup and Configuration: COUGAAR 9.4

Overview

To install COUGAAR 9.4:
Go to the cougaar website (www.cougaar.org) to get the latest windows Cougaar installer which will install Cougaar as well as a MySQL database.  This will allow users to run off of a local MySQL database rather than accessing an Oracle database located in Cambridge, MA at BBN Technologies.

Or you may install Cougaar 9.4, without mySQL, by unzipping the cougaar.zip and cougaar-support.zip  The rest of the zip files are not required to run Cougaar. 

The zips are designed to be unpacked in the same directory, e.g. on
unix:
        mkdir cougaar
        cd cougaar
        unzip /tmp/cougaar.zip
        unzip /tmp/cougaar-support.zip
        setenv COUGAAR_INSTALL_PATH `pwd`

The zips are intended to be installed in the same directory. They each have a Manifest directory with a "zipfile.version" (e.g., cougaar-support.version) which contains something like:
COMMENT = cougaar thirdparty jars
REPOSITORY_TAG = V9_4_0                          # branch of repository
REPOSITORY_TIME = 07/01/2002 13:00:29          # checkout time of repository
ARCHIVE_TIME = 07/01/2002 13:11:03               # zip time
All your zips should have the same REPOSITORY_TAG and REPOSITORY_TIME.

To verify correct installation of the COUGAAR software, a small but non-trivial test society relying only on ALPINE Plugins is included in "configs/minitestconfig." This society is configured to run in a single node on a single computer. This configuration can be modified as required for Plugin testing.  Minitestconfig features a reduced 4-cluster society with GLS initiating at 3ID and flowing down to subordinate clusters. The ARBN cluster generates Transport tasks that are eventually allocated to MCCGlobalMode. These tasks are allocated and the allocation results propagated back to the task originator. The task originator (ARBN) evaluates the results and modifies the task preferences to achieve an acceptable solution. Subcontractors are encouraged to bring their Plugins up to full compliance with the new infrastructure modifications, possibly using the minitestconfig society. A separate document provides a functional description of the minitestconfig society.

CSMART - Cougaar Society Monitoring, Analysis and Reporting Tool is the preferred way of running a Cougaar society. See CSMART Getting Started. In particular, this document also gives a thorough step-by-step instruction for installing Cougaar and getting started running Cougaar.

The following "jar" files are used in the baseline minitestconfig society configuration and are located in the cougaar\lib and cougaar\sys directories:
    cougaar/lib

  • bootstrap.jar: COUGAAR bootstrapper classloader
  • util.jar: Cougaar component architecture
  • core.jar: COUGAAR core infrastructure
  • glm.jar: domain infrastructure support and Logistics Model Plugins
  • toolkit.jar: standalone utilities tools
  • build.jar: code generators for COUGAAR software
    cougaar/sys
  • xerces.jar: XML parser
  • log4j.jar: logging
  • servlet.jar: java servlet mechanism
  • tomcat_40.jar: tomcat webserver
  • jcchart451K: for some UIs in glm
  • fesi-111.jar: only if using Fesi generic plugin
  • one of the three jdbc drivers: classes12.zip, mm-mysql-2.jar or bbn-postgresql7.1.3.jar

Installation and Configuration

Please read the following information carefully.

Unpacking the release packages (zip files) should produce the following directory structure inside your Cougaar installation directory:

  • lib ---------------- "jar" files for infrastructure, Plugins, and other custom COUGAAR components
  • sys --------------- "jar" and "zip" 3rd party software used by the infrastructure and Plugins
  • bin ---------------- "bat", "csh", and "sh" scripts for setting environment variables and paths and starting applications
  • configs ----------- subdirectories of society configuration files (e.g., minitestconfig)
  • doc ------ Release Notes (index.html) and other documentation for COUGAAR infrastructure.
  • <product>------ product specific data files and non-java executables (each Plugin or COUGAAR participant will have a corresponding subdirectory). These include the basic Cougaar modules such as core, glm, aggagent, webtomcat, and csmart.
For further details see the corresponding section in the Release Notes.
The following information describes the configuration scheme for Cougaar software distributions. Please note the new organization of release software and data files for Plugin software and society configuration directories.
  1. There are environment variables and Java system properties as defined in the table below. The purpose of these parameters is to control COUGAAR system behavior and minimize placing environment variables in the Node.bat or Node.csh files.
  2. The environment variable COUGAAR_INSTALL_PATH must point to the directory in which the Cougaar software was unzipped . This is used to set the (java) system property org.cougaar.install.path which must point to the same directory. These variables will allow the infrastructure and Plugins to find the necessary "jar" files (lib and sys) and data files (<product>).
  3. The environment variable COUGAAR_DEV_PATH may optionally be set to point to a directory or jar file of custom developed components which you wish to be available at runtime.
  4. You may also optionally set the environment variable COUGAAR_WORKSPACE to point to the directory in your installation where runtime components should write output. This defaults to COUGAAR_INSTALL_PATH/workspace.
  5. For system testing in a single node configuration, "serialization" of COUGAAR objects can be forced by setting the org.cougaar.core.message.alwaysSerialize system property to true. This feature cannot be used in a multiple node scenario and should be false.
  6. Although we have not tested all society configurations on all types of processing platforms, all COUGAAR infrastructure software is compatible with NT, UNIX and LINUX operating systems.
  7. The Cougaar bootstrapper will find all classes in jar files in the $COUGAAR_INSTALL_PATH/lib and $COUGAAR_INSTALL_PATH/sys directories. For your convenience, place additional 3rd party jars in the latter.
  8. Execution of "Node.bat" and "Node" (csh) requires passing the "node name" as the first argument to these scripts, rather than coding them into the scripts. The node name argument must match the node name used for the <nodename>.ini file.
  9. To find initialization files (e.g., clustername-prototype-ini.dat and OPlan.ini), the infrastructure will first look in the local directory and then look in the ${COUGAAR_INSTALL_PATH}\configs\common.


Table 1: Environment variables and system properties used to control COUGAAR system behavior
 
 
Name Type Value Usage

org.cougaar.core.message.alwaysSerialize Java System Property Boolean (true or false) Forces serialization of objects between clusters. Only usable in single node configurations
org.cougaar.core.agent.enforceTransactions Java System Property Boolean (True or False) Produces warnings when LogPlan accesses are attempted outside safe transaction boundaries

Populating Agents with Assets

The initial population of assets for a particular organization is handled by Data Plugins, specifically the LDMXMLPlugin  that reads assets from an XML formated file. Also included in the release is an LDMSQLPlugin that reads assets from Cougaar domain database. On start-up, the XML Plugins query the XML files and create the specified LDM objects.

The JDBC drivers required for the LDMSQLPlugins are freely available.  Please refer to the Heterogeneous Database Capability document to read about Cougaar running with Oracle, MySQL and PostgreSQL databases.
