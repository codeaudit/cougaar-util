COUGAAR Society Startup and Configuration: COUGAAR 8.6

Overview

To install COUGAAR 8.6:
Go to the cougaar website (www.cougaar.org) to get the latest windows Cougaar installer which will install Cougaar as well as a MySQL database.  This will allow users to run off of a local MySQL database rather than accessing an Oracle database located in Cambridge, MA at BBN Technologies.

Or you may install Cougaar8.6, without mySQL, by unzipping the cougaar.zip and cougaar-support.zip  The rest of the zip files are extras.  Furthermore, the cougaar-support contents do not change very often, so you can unzip that once and keep the contents as updates occur to cougaar.zip.

The zips are designed to be unpacked in the same directory, e.g. on
unix:
        mkdir cougaar
        cd cougaar
        unzip /tmp/cougaar.zip
        unzip -o /tmp/cougaar-support.zip
        setenv COUGAAR_INSTALL_PATH `pwd`

Although the zips are intended to be installed in the same
directory, they all have a top-level file "version.txt" which contains
something like:
COMMENT = cougaar thirdparty jars
REPOSITORY_TAG = V8_6_0                          # branch of repository
REPOSITORY_TIME = 11/01/2001 13:00:29          # checkout time of repository
ARCHIVE_TIME = 11/01/2001 13:11:03               # zip time
All your zips should have the same REPOSITORY_TAG and REPOSITORY_TIME.
unzip will ask you if you want to overwrite version.txt -  do what you
wish.  Each generated jar file also gets this information, btw.
You can unpack all the zips into the same directory without warnings
using the right argument to unzip.  on unix "unzip -o" will do the trick.

To verify correct installation of the COUGAAR software, a small but non-trivial test society relying only on ALPINE PlugIns is included in "alpine\configs\minitestconfig." This society is configured to run in a single node on a single computer. This configuration can be modified as required for PlugIn testing.  Minitestconfig features a reduced 4-cluster society with GLS initiating at 3ID and flowing down to subordinate clusters. The ARBN cluster generates Transport tasks that are eventually allocated to MCCGlobalMode. These tasks are allocated and the allocation results propagated back to the task originator. The task originator (ARBN) evaluates the results and modifies the task preferences to achieve an acceptable solution. Subcontractors are encouraged to bring their PlugIns up to full compliance with the new infrastructure modifications, possibly using the minitestconfig society. A separate document provides a functional description of the minitestconfig society.

The following "jar" files are used in the baseline minitestconfig society configuration and are located in the cougaar\lib and cougaar\sys directories:
    cougaar/lib

  • core.jar: COUGAAR core infrastructure
  • glm.jar: domain infrastructure support and Logistics Model PlugIns
  • toolkit.jar: standalone utilities tools
  • build.jar: code generators for COUGAAR software
  • planserver.jar: tools for extracting, entering, and displaying information from COUGAAR
    cougaar/sys
  • xerces.jar: XML parser
  • log4j.jar: logging
  • jcchart451K: for some UIs in glm
  • fesi-111.jar: only if using Fesi generic plugin
  • one of the three jdbc drivers: classes12.zip, mm-mysql-2.jar or bbn-postgresql7.1.3.jar

Installation and Configuration

Please read the following information carefully. The following information describes the current organization of "jar" and data files and system parameters. In addition, this information describes the organization of PlugIn software submissions and how PlugIns must reference private data files and software libraries.

Unpacking the release packages (zip files) should produce the following directory structure inside the new alp-<builddate> directory:

  • lib ---------------- "jar" files for the infrastructure, ALPINE PlugIns, and COTS libraries
  • sys --------------- "jar" and "zip" 3rd party software used by the infrastructure and PlugIns
  • bin ---------------- "bat" and "csh" scripts for setting environment variables and paths
  • configs ----------- subdirectories of society configuration files (e.g., minitestconfig)
  • src --------------- infrastructure source code (in zip files)
  • doc ------ Release Notes (release_notes_index.html) and other documentation for COUGAAR infrastructure.
  • api ------"javadoc" for COUGAAR infrastructure
  • <product*>------ product specific data files and non-java executables (each PlugIn or COUGAAR participant will have a corresponding subdirectory)
The following information describes the configuration scheme for Cougaar software distributions. Please note the new organization of release software and data files for PlugIn software and society configuration directories.
  1. There are environment variables and Java system properties as defined in the table below. The purpose of these parameters is to control COUGAAR system behavior and minimize placing environment variables in the Node.bat or Node.csh files.
  2. The environment variable COUGAAR_INSTALL_PATH must point to the installation directory "cougaar". This is used to set the (java) system property org.cougaar.install.path which must point to the same directory. These variables will allow the infrastructure and PlugIns to find the necessary "jar" files (lib and sys) and data files (<product-n>).
  3. For system testing in a single node configuration, "serialization" of COUGAAR objects can be forced by setting the org.cougaar.core.message.alwaysSerialize system property to true. This feature cannot be used in a multiple node scenario and should be false.
  4. Please note that the Node.bat/csh files are only present in the cougaar/bin directory. They have been removed from all configuration directories to encourage use of a single version in a standard location. We suggest adding %COUGAAR_INSTALL_PATH%\bin to your PATH environment variable.
  5. Although we have not tested all society configurations on all types of processing platforms, all COUGAAR infrastructure software is compatible with NT, UNIX and LINUX operating systems.
  6. The file "setlibpath.bat" in "core\bin" creates the path to all "jar" files and "setarguments.bat" defines the command line arguments to  java.exe, rather than using "node.bat" to perform this function. All "node.bat" files should call these files.
  7. Execution of "Node.bat" and "Node" (csh) requires passing the "node name" as the first argument to these scripts, rather than coding them into the scripts. The node name argument must match the node name used for the <nodename>.ini file.
  8. To find initialization files (e.g., clustername-prototype-ini.dat and OPlan.ini), the infrastructure will first look in the local directory and then look in the {$COUGAAR_INSTALL_PATH}\configs\common.


Table 1: Environment variables and system properties used to control COUGAAR system behavior
 
 
Name Type Value Usage
COUGAAR_INSTALL_PATH Environment Variable <drive>:\alp-<builddate> Top-level location of COUGAAR software 
org.cougaar.install.path Java System Property <drive>:\alp-<builddate> Top-level location of COUGAAR software 
org.cougaar.core.message.alwaysSerialize Java System Property Boolean (true or false) Forces serialization of objects between clusters. Only usable in single node configurations
org.cougaar.core.cluster.enforceTransactions Java System Property Boolean (True or False) Produces warnings when LogPlan accesses are attempted outside safe transaction boundaries 
org.cougaar.core.message.retryInterval Java System Property Integer Timeout period for messages in milliseconds 
org.cougaar.core.message.retryMaxCount Java System Property Integer Max number of times to retry failed messages 
org.cougaar.domain.foo Java System Property Class Name Specifies the class that defines the domain package where the domain corresponds with "foo"
org.cougaar.cluster.startTime Java System Property <date> Note this is set to 08/10/2005 by default for optimal use of the small-135 society and Oplan. Specifies the society start time.

Populating Clusters with Assets

The initial population of assets for a particular organization is handled by Data PlugIns, specifically the LDMXMLPlugIn  that reads assets from an XML formated file. Also included in the release is an LDMSQLPlugIn that reads assets from the JTAV snapshot database on eiger. Examples can be found in the alpine.plugin package. On start-up, the XML PlugIns query the XML files and create the specified LDM objects.

The JDBC drivers required for the LDMSQLPlugIns are freely available.  Please refer to the Heterogeneous Database Capability document to read about Cougaar running with Oracle, MySQL and PostgreSQL databases.
