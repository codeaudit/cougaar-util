Overview of this Release

Build 11.0 is a major release, including multiple new pieces of functionality.

  • Updated Jena to version 2.0 and Xerces to 2.6.0:
    • The new Jena jar required an update to icu4j.jar, and the addition of antlr.jar, jakarta-oro-2.0.5, and rdf-api-2001-01-19
    • The updated Xerces changed names to xercesImpl and xml-apis.jar (split implementation from APIs).
    • Servicediscovery code was changed to support the new Jena. Additionally, we use the built-in Jena file-caching. We still list the cached files in proxyTable.txt, but this is not just documentation. We no longer use custom caching. Additionally, Time.daml was added to the set of cached files.
    • Jena2 now uses log4j. To turn down Jena WARNings, in particular from the RDFDefaultErrorHandler, specify the log4j property "log4j.category.com.hp.hpl.jena=ERROR".
  • GLS tasks may no longer be rescinded. For each Oplan stage, the GLSInitServlet now sends a new GLS task with the additional oplan stage - leaving the old tasks behind. It is generally bad usage to modify a task in place to have entirely different meaning. In this case, it appeared to cause confusion about the meaning of the allocation results. See bug 3407.
  • The MTS implementation has been factored into a separate module and project. In CougaarForge, it is now the "std" module of the "mts" project. Cougaar builds package it as "mtsstd.jar". The major effect of this refactoring is that most of the classes previously in org.cougaar.core.mts are now in org.cougaar.mts.std (e.g. those which implement the MTS interface have changed packages, while those which define the interface have not. See ReleaseNotes/11.0/mts-refactor.txt for details. Also see the perl script ReleaseNotes/11.0/mts-refactor.pl which will do most of the substitutions requires to port 10.0 code to 11.0. As a side effect of the refactoring, there is a new service available at the node level SocketFactoryService which allows clients to find an appropriate SocketFactory without needing to know about the MTS implementation which may be supporting it.
  • The uiframework no longer requires the non-open source datagrabber module for compilation. Actual usage of some pieces will likely still require that software.
  • Task.getPreferredValue now returns Double.NaN instead of -1.0 when the value is undefined (cbug 2919).
  • The base unix shell scripts have been greatly simplified. setarguments and setlibpath are gone, fully replaced by the boost script (now used by the rest of the scripts). Node now fully subsumes the functionality of the now deleted XMLNode and PNode scripts.

Build 10.4.1 is a minor release, including several bug fixes not ready in time for 10.4.

  • Database loading scripts have been simplified. As a result, you WILL need to update your cougaar.rc file. See the Data Access documentation for details on this change. See the or the CSMART Install document for details on the new simplified database load process.
  • ClusterImpl has been dropped entirely to avoid some confusing situations in the future.
  • Dropped the scalability module, which has not been functional for some time.
  • OPLAN and GLS propagation mechanisms have been significantly modified. Users of the GLSInitServlet will need to update to match the new API. The provided GLSInit UI will look different. To use it: 1) Wait for all superior/subordinate relationships to be established. 2) Hit "Connect" to talk to the GLSInitServlet, 3) Hit Get Oplan Info, then modify the C-0 date as desired, and 4) Send the GLS Task (which also instructs each agent to separately retreive its OPLAN.
  • The searchCommunity, getCommunity, and getParentCommunities methods in CommunityService now return results if the required information is available in local cache. Callbacks are still used for those operations requiring interaction with remote manager agent. WhitePages timeout values may now be set via the org.cougaar.community.manager.WPTimeout property. Default value is 60000 (ms).
  • The default set of domains now includes the ServiceDiscovery domain in the LDMDomains.ini file. ServiceDiscovery is now used to establish all provider relationships in more complex (UltraLog) configurations.
  • Increased the default max Threads in the Cougaar Thread Pool from 30 to 100.
  • Multiple fixes to the ServiceDiscovery module, including changes to simplify installation and usage. See servicediscovery/doc/README.txt for installation and usage instructions
  • Bug 2558: Add @event javadoc taglet handling in builds.
  • Enhancement to GenericTablePlugin (bug 2914)
  • Some modifications to the DemoTimeControlService and ClusterServesClocks APIs in preparation for enhancements to this functionality.
  • Several WP bugs have been fixed, such as better startup support to avoid version number lookup problems (bug 2963) and new behavior to deconflict overlapping local rebind requests (bug 2902).
  • Error-checking code has been added to SimpleAgent to verify that the agent name has been correctly set.
  • Agent mobility "add agent" support has been fixed to specify SimpleAgent instead of ClusterImpl when creating new agents, plus it now allows the client to specify the agent's full ComponentDescription (bug 2924)
  • Otherwise unhandled messages to an agent is held during agent boot until it has had a chance to load all possible MessageHandlers. This prevents a problem where YP messages could start flowing before the YPServer was loaded. (bug 2926)
  • The servlet server was fixed to catch a new variant of "port already in use" exception now thrown by some JVMs (bug 2921)
  • WP blocking calls removed in MTS and Metrics Service (bug 2925)
  • Metric Service has new servlets to allow remote query of metrics via http. Both an XML and Java format is supported.(2776)
  • Metrics Service has support for message flow between agents. The Metrics service can be queried for AgentFlow paths, but 10.4.1 does not actually collect the statistics. Statitistics collection Robustness UC3 and will be moved of 10.4.2. See the Metrics Service web-based documentation
  • The Agg Agent code was fixed so that it could handle receiving a null reference to the LoggingService. This has been fixed so that it will work properly in the case of no LoggingService. (bug 2969)
  • A new servlet, org.cougaar.core.plugin.freeze.FreezeServlet , can be use to suspend the activity of a single agent. It accepts the same rule parameters as the current FreezePlugins. See bug 2968
  • GLM Constants class may now be sub-classed.
  • XMLComponentInitializer now does some sanity checks on the input Node name and society XML files.
  • Bug 2855 -- In addition to preferences, task prepositional phrases, contexts, and verb are compared for determining if a task has changed. This fixes a problem of plugins not seeing such changes.
  • The name of the RemoteClusterAllocationLP has been changed to RemoteAllocationLP.
  • Bug 2889 -- Tasks with end time preferences that are in the past are not propagated to other agents. This is done to avoid resurrecting deleted tasks after a restart. Some plugins create tasks without regard to whether the tasks have end times in the past and may encounter problem. The "past" ends 24 hours before the present scenario time.
  • Fixed bug 2923: Correctly handle Log4J config params prefixed by "org.cougaar.util.log". Also, if a -D logging config param specifies a file for a File appender in a non-existent directory, try to create it. If that fails, skip this property.

  • YPServer bug 2962 fix to ensure early incoming messages are handled correctly.
  • Minitestconfig now uses differently named GLSMiniAllocator and GLSMiniExpander, as the originals now support finding relationships and oplan to support ServiceDiscovery. Functionality is the same.
  • Fixed bug 3001 (and 2993): the YPService now queues messages at both client and server ends for handling by pooled threads. This avoids a potential deadlock when using the blackboard-based yp queries while persisting.

Build 10.4 is a major release, including multiple new features and some repackaging. It incorporates all new functionality introduced in the experimental 10.3 release, along with the following additional features and fixes.

  • UltraLog specific configuration information has been removed from the open source Cougaar release. This simplifies basic Cougaar installation and slims the download. However, there is not longer a complex configuration utilizing all the power of Cougaar supplied with the Cougaar distribution.
  • Several obsolete modules have been removed from the Cougaar distribution, including the blackjack module, and the dbms module.
  • The CSMART Experiment controller has been refactored for improved performance and readability. Additionally, the controller is now capable of running standalone, and reading Experiment definitions from XML files that conform with core/configs/common/society.xsd
  • The refconfig db (used with XML societies) is now loaded by a script: csmart/data/database/scripts/mysql/load_ref_data.sh
  • The RMI-based JNDI Naming Service has been replaced with an in-band White Pages implementation modeled after DNS. Minor API changes to the WP service have been implemented (e.g., to classes AddressEntry, Application). For design details see the Cougaar Architecture Document. The behavior of the new WP should be identical to the old Naming Service.
  • Minitestconfig now includes the WhitePages servlet. It also uses the newer GLM completion servlet instead of the older one.
  • The minitestconfig directory now includes an XML file definition of the society configuration. To run this, create the refconfig database (following the CSMART install instructions), and use the XMLNode script.
  • bin/Node is now implemented as a sh script rather than csh. It is much more flexible and avoids needing to modify the script to pass extra options in most cases.
  • Significant cleaning up (mostly removal) of long-unused code and files. If you discover that your favorite package has been removed in this release, please write to cougaar-developers@cougaar.org
  • Deprecated methods in TimeSpan.java have been removed. See Bug 188. The blackjack module has some callers of these methods thus if this module is required in the future, it needs to be updated for the replacement methods.
  • Some unused 3rd party jar files were removed.
  • OrgActivity has been refactored to separate interface from implementation and factory.
  • The default society start time is now 7/11/05 (vice 8/10) (bug 2883).
  • GLSInitServlet/GLSClient and GUI have been changed. 'Publish Oplan' button has been replaced by 'Get Oplan Info'. Pressing this button retrieves some Oplan information but does not publish the oplan to the agents. A default C0 date can now be overridden via the GUI. GLS Task now contains additional information (oplan id and C0 date) and sending GLS causes each agent to read the database for the oplan and org activities.
  • A new unified source view script has been added to this release (Bug #2502). See build/bin/unify_source.sh for details.
  • The Thread service has a new Top Servlet for viewing the currently running and queued schedulables. This servlet is useful for debugging threading issues where threads do unnecessary or buggy sleeps, waits, or blocks on IO. In general, threads in plugins should NOT use sleep, wait, or blocking IO. See the Thread Service web-based documentation
  • The Metrics service provides valuable information for debugging and monitoring a society. The Metric Service has been improved with better servlets for viewing metrics. A new Gossip service has been added. The new services and servlets are pluggable components that can be added to the configuration. See the Cougaar Developers Guide for details.
  • The default society start time is now 7/11/05 (vice 8/10) (bug 2883).
  • The interface "DomainBindingSite" has been removed. Domains should now use the plain BindingSite API. New domain-internal services have been created (DomainRegistryService, XPlanService). This may impact domain developers (cf. Bug #1375).
  • The ContainerSupport base class has been modified to support future work on the component model viewer. Developers of containers should see Bug #1113 for details.
  • The "doc/details" and "doc/ComponentModel" directories have been removed. Their content has been merged into the Cougaar Developers Guide or moved to "doc/ReleaseNotes/[release number]".

Build 10.3 is a minor release, including a few new features and deprecations.

  • The Logging API setup has been changed somewhat. Most users will not see any changes, but the properties which specify the logging configuration file to use (e.g. org.cougaar.util.log.config) are interpreted using a stricter set of rules. In particular, it is no longer merely passed as-is to the ConfigFinder. Instead, it is interpreted as a URL or as a filename relative to COUGAAR_INSTALL_PATH. For backwards compatibility, if there are no "/" characters in the name, it will be interpreted relative to configs/common instead. If the name ends in .xml, it will be parsed using xml4j's DOMConfigurator. If it ends in ".props" or ".properties", by the PropertyConfigurator. If no file is specified or found, it will use a simple default configuration.
  • The example "small-135" configuration has been removed. It was no longer functional or supported. For complex society examples, see the 1AD societies supplied with the CSMART module.
  • The YP Client API has changed considerably to support non-blocking queries. Also now supported are blackboard-based YP interactions. YP no longer requires a modified UDDI4J jar file. The YP server code now supports multiple servers in the same VM.
  • ServiceDiscovery now uses YP. See servicediscovery/doc/yp-info.txt for more information.
  • The method for obtaining the oplan has changed. Each agent now obtains the oplan and its org activities directly from the database. The GetOplanPlugin has been replaced by the OplanReaderPlugin and the GLSInitServlet has been modified. The OplanEditServlet must be loaded at each agent in order to modify Org Activities. This change should be transparent to most users.

Build 10.2 is a minor release, including a few new features and deprecations.

  • ComponentDevelopers should try not to put runtime data or configuration files in configs/common. Instead, these files should be stored in the local module for that Component. Additionally, configuration files should be kept separate from data files. To encourage this, the ConfigFinder may now be created with an optional "module" argument, as in: ConfigFinder.getInstance("albbn");. This argument refers to the name of a directory under Cougaar_Install_Path. When supplied, the ConfigFinder will add 4 directories local to that module to the front of the search path: $CIP/$MOD/configs/$CONFIG;$CIP/$MOD/configs;$CIP/$MOD/data/$CONFIG;$CIP/$MOD/data. See the class JavaDoc for details.
  • Used the new above methods, and moved CSMART related .q files to csmart/data/common. The preferred location for recipeQueries.q is now csmart/data/common, however the old location will still work.
  • Remove unused plugins: AssetDataPluginFile, AssetDataPluginDB, OrgDataPluginFile, and OrgDataPluginDB. Additionally, removed AssetDataPlugin (unused) and replaced it with the renamed AssetDataPluginBase.
  • Bug 1611: Cleaned up Node initialization. Note that all changes are completely backwards compatible. By specifying a system property, org.cougaar.core.node.InitializationComponent, you can specify how component configurations are obtained. "File" is shorthand for INI files, "DB" is shorthand for the CSMART database, and "XML" allows running from XML files (only minimally supported so far). The default Asset (Entity or Organization) Initialization service will initialize from the CSMART DB if the Node did, from a new reference database if the Node initialized from XML (only minimally supported), otherwise from INI files as before. However, users may specify a new component at BINDER priority that provides an AssetInitializerService, to use other behavior. Additionally, Communities are initialized from the CSMART DB if the Node was, otherwise from the (XML) file. Again, an alternate may be provided at BINDER priority. Note that CSMART now provides this -D argument in all Nodes by default, indicating that Nodes should completely initialize from the CSMART database.
  • Began minimal support for initializing Cougaar societies from XML files. Configurations should follow the schema in core/configs/common/society.xsd. Start Nodes from XML using bin/XMLNode.[bat/sh]. For instructions on running a sample 1AD society, see configs/ul/Readme.txt.

Build 10.0 is a major release, including significant code repackaging and cleanup.

Please see the porting notes for documentation on how to port Cougaar 9.4 code to Cougaar 10.0.

  • A new Cougaar FAQ and Package Map are now available with the release.
  • ClusterIdentifier has been replaced by MessageAddress, and static MessageAddress factory methods have been added (bugs 1116, 1325, 2496). The included fixCID script can be used to help automate the porting of pre-Cougaar-10.0 source code.
    As part of this change, the CSMART configuration database has changed. We recommend reloading your database, following the CSMART install instructions. You may also update your database in-place, using the provided update script in csmart/data/database/scripts/mysql/sql/cluster-id-removal.sql Finally, at experiment load time, if CSMART still sees a ClusterIdentifier in a PropertyGroup, it will replace it with a MessageAddress. You will need to re-save your experiment though.
  • AspectValues have been refactored to be immutable and support static factory methods (bugs 2430 and 2529). The included fixAV script can be used to help automate the porting of pre-Cougaar-10.0 source code.
  • The planning domain has been separated from the core and a new "planning.jar" is now included in the release. Several classes have been assigned new package names to better fit into core (org.cougaar.core.*) or planning (org.cougaar.planning.*). The included core-repackage script can be used to help automate the porting of pre-Cougaar-10.0 source code.
  • As part of the core/planning split, the Cougaar Domain APIs have been significantly refactored. Domain developers should see the domain porting notes.
  • The use of custom BindingSites has been almost entirely removed (bug 1375). The PluginBindingSite API in particular has been removed, which means that plugins that extend ComponentPlugin (which extends BlackboardClientComponent) must use the plugin's "getAgentIdentifier()" and "getConfigFinder()" methods instead of casting the binding site. The included fixBindingSite script can be used to help automate the porting of pre-Cougaar-10.0 source code.
  • Related to the above BindingSite refactoring, most of the old "XServesY" interfaces, such as "ClusterServesPlugin" and "LogPlanServesLogicProvider", have been replaced with equivalent Service-based APIs.
  • The XMLizable marker interface has been removed. The "/tasks" servlet's XML encoder now assumes that all blackboard objects can be inspected (via reflection/BeanInfo). This simplifies most blackboard data structures, since they were found to universally implement the default XMLizable behavior.
  • PSP support has been removed and officially replaced by servlets (bug 1017). All PSPs have been removed (most were already ported to equivalent servlets). Most of the MLM-UI code, which depended upon PSPs, was also removed (org.cougaar.mlm.ui.*) except for the TPFDD data grabber.
  • The initial WhitePagesService, which will replace the current NamingService, has been defined and a JNDI-based implementation has been implemented. Please see the Cougaar Developers' Guide for details.
  • The servlet server has been updated to use the new white pages service instead of the JNDI naming service. A new "/wp" servlet (org.cougaar.core.util.WhitePagesServlet) has been added to view the contents of the white pages. The built-in "/agents" servlet has been enhanced to traverse the naming hierarchy instead of listing all agent names.
  • The "/tasks" servlet has been enhanced to use the "/agents" servlet for its drop-down selection of agent names, a pattern that will be required for future servlets. Servlets should no longer use the "getAllEncodedAgentNames()" utilities, since these will not scale.
  • Insertion points for the major components are now defined by public constants in the appropriate classes. For example, the insertion point for an Agent is org.cougaar.core.agent.Agent.INSERTION_POINT (Plugin is org.cougaar.core.plugin.PluginBase). Use these constants wherever possible.
  • Message transport was updated to use new white pages naming services and topology service. The LinkProtocol API has changed slightly as a result.
  • Made multiple CSMART Changes. As usual, see the CSMART Release Notes for details.
    • Removed various obsolete and unused pieces of code. This includes the CSMART Domain (change your LDMDomains.ini as necessary), the ABC Impacts and society, and the un-used multiple trial support.
    • Replaced the 'Default Workspace.bin' workspace file with a simple XML file.
    • Changed to use SimpleAgent by default, rather than ClusterImpl, and updated to use MessageAddress instead of ClusterIdentifier. Reload your database to match.
  • Support for dynamic service discovery has been added through the servicediscovery module. The default society configurations still use the static client/provider relationships defined by the CSMART config database. If you would like to use dynamic service discovery, see the COUGAAR Service Discovery Installation Guide for details.

Organization of a Release 

When you download a Cougaar release, there are several logical pieces.
  • Cougaar runtime files: these come from the cougaar.zip file and include jar files, startup scripts, and data files. These in turn are spread out across several directories or "modules". The directories that come in cougaar.zip do not contain source code or Javadoc, which is available separately.
    • lib/: This directory contains the Cougaar-developed jar files.
    • bin/: This directory contains all the primary Cougaar startup scripts.
    • doc/: This is where Cougaar documentation is stored, including this file. Note that the Developer's Guide and Javadoc come separately.
    • configs/: All the basic runtime data and configuration files are stored here. In various documents you will often see reference to "configs/common", which subdirectory is found here, and includes common data files. Some simple flat-file base Cougaar society definitions are also found here.
    • aggagent/: This software module includes the Aggregation Agent architecture for collecting data from various Cougaar Agents and summarizing it.
    • bootstrap/: Cougaar classloader that ensures jar files in usual places are included, without requiring CLASSPATH modifications.
    • build/: The scripts used to compile and build a Cougaar release.
    • community/: Support for defining communities of Agents and their communitications.
    • contract/: Plugin contracts - not completely supported.
    • core/: The core Cougaar Agent architecture.
    • csmart/: CSMART is the Cougaar Society Management, Analysis, and Reporting Tool. Use it to configure, run, and monitor large Cougaar societies. See csmart/doc/index.html
    • datagrabber/: For collecting and displaying plan results, in particular TPFDD (military logistics) displays.
    • glm/: Defines Plugins and data model for a military logistics domain and application.
    • javaiopatch/: Small JDK over-rides to support persistence.
      • application.
    • qos/: Quality of Service support for Message Transport
    • quo/: More Message Transport support
    • server/: The AppServer (or Node Server) is a daemon used by CSMART to remotely startup and stop Cougaar Nodes.
    • toolkit/: Various support utilities, particularly to support scheduling.
    • uiframework/: Framework for constructing some UIs.
    • util/: Cougaar component architecture and utility classes, including logging.
    • vishnu/: GA based scheduler.
    • webserver/: Servlet support.
    • webtomcat/: Integration with Apache Tomcat.
  • Cougaar source: The source code for each of the various Cougaar modules, along with Javadoc, is available in <module>-dev.zip. Here you will also often find JUnit regression tests (in a sub-directory called "regress"), and examples (in the "examples" sub-directory). Of particular note is tutorial-dev.zip which has the sources for a number of simple tutorials for learning Cougaar, plus examples, as used in the periodically offered Cougaar Tutorial classes (from tutorial-dev.zip).
  • Consolidated core API: The cougaar-api.zip contains the consolidated Javadoc for the "basic" Cougaar modules, for extraction into the top-level doc directory.
  • Cougaar third-party support libraries: The cougaar-support.zip file contains third-party jar files and libraries, which when extracted go in the sys/ directory.
  • md5sums.txt:  md5 checksums of all the zips.
  • build.out: output of the build process
  • errors.out: Any errors or warnings encountered by build.
To install Cougaar:
Go to the cougaar website (www.cougaar.org) to get the latest windows Cougaar installer which will install Cougaar as well as a MySQL database.  This will allow users to run off of a local MySQL database.

Note that if you are installing MySQL on linux or unix, you must instruct the database to translate all table names to lowercase (which is the default on Windows machines).  To do this, you must set the variable lower_case_table_names=1 by including the line in /etc/my.cnf:

[mysqld]
set-variable   =   lower_case_table_names=1

Alternatively you can follow these instructions.

  1. You'll need cougaar.zip and cougaar-support.zip (the first time) to get a runnable system.  The rest are extras.  Furthermore, the cougaar-support contents do not change very often.
  2. The zips are designed to be unpacked in the same directory, e.g. on unix: 
            mkdir cougaar 
        cd cougaar 
        unzip /tmp/cougaar.zip 
        unzip -o /tmp/cougaar-support.zip 
        setenv COUGAAR_INSTALL_PATH `pwd`

Note that $COUGAAR_INSTALL_PATH/bin/Node and setarguments.bat set the Cougaar start time with the following Node argument:
-Dorg.cougaar.core.cluster.startTime=07/11/2005

For instructions on installing and using CSMART, see the CSMART installation document.

Test Harness Example Implementations

We only maintain society configurations that we are currently using for testing purposes and for  Cougaar demonstrations. These society configurations are available in subdirectories under "configs" in the release. To execute these Cougaar societies, these nodes can be executed on a single computer or distributed across multiple computers. 

One society, minitestconfig will work "out of the box". We intend to expand this test society in subsequent releases in size and complexity to test new infrastructure features and to provide reference examples of components component and service design patterns. Plugin and component developers are encouraged to use minitestconfig as a starting point for testing their software for compliance with the latest infrastructure release. Minitestconfig can be run as a multiple node, multiple host society. 

Delivering Software Products 

All Plugin software must use the Java system parameter "org.cougaar.install.path" as the basis for all searches, data files, etc. For writing output, use "org.cougaar.workspace". For example, if your product is "tops," your classes will be in: 

"$COUGAAR_INSTALL_PATH\lib\tops.jar"
and your java code could find your data files with: 
String path = System.getProperty("org.cougaar.install.path") +
              File.separatorChar + "tops" +
              File.separatorChar + "myData.dat";
